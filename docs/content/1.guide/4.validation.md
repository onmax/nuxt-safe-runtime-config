The module validates your runtime config to catch configuration errors early. By default, validation runs at build time only.

Validation is handled by the package's Nitro module. In Nuxt apps, the Nuxt module exposes the user-facing API, types, and imports, then passes validation options into Nitro. Generated validation files are internal implementation details and their paths are not part of the public API.

## Build-time Validation (Default)

With default settings, build validation runs through Nitro during:

- **Dev start** - when running `nuxt dev`
- **Build** - when running `nuxt build`

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  safeRuntimeConfig: {
    $schema: runtimeConfigSchema,
    validateAtBuild: true, // default
  },
})
```

Build-time validation uses your schema library directly. It catches:

- Missing required values
- Type mismatches
- Invalid formats

Inline schemas provide these build-time benefits:

- Zero runtime overhead
- Fail fast before deployment
- CI/CD friendly
- The validation library does not need to run in production

## Runtime Validation (Opt-in)

Runtime validation catches issues that only appear in production, such as environment variables that differ from their build-time values. Inline schemas retain JSON Schema validation:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  safeRuntimeConfig: {
    $schema: runtimeConfigSchema,
    validateAtRuntime: true,
    onError: 'throw',
  },
})
```

When you enable runtime validation with an inline schema, the module:

1. Generates JSON Schema from your Standard Schema.
2. Validates `useRuntimeConfig()` through a Nitro plugin on cold start.
3. Uses [@cfworker/json-schema](https://github.com/cfworker/cfworker/tree/main/packages/json-schema), an edge-compatible runtime validator.

JSON Schema describes and validates data shapes, but it cannot execute Standard Schema transformations. Use a schema module when runtime input and validated output have different types.

::important
Pass a string `$schema` path when runtime behavior depends on transformations. Inline schemas validate the runtime shape without executing transforms.
::

## Transform Runtime Values

Nitro environment overrides can arrive as strings even when your application expects booleans or numbers. Define input coercion once in the schema module so every consumer receives the same validated output:

```ts [runtime-config.schema.ts]
import * as v from 'valibot'

const runtimeBoolean = v.pipe(
  v.union([v.boolean(), v.picklist(['true', 'false'])]),
  v.transform(value => value === true || value === 'true'),
  v.boolean(),
)

const runtimeNumber = v.union([
  v.number(),
  v.pipe(v.string(), v.decimal(), v.transform(Number), v.number()),
])

export default v.object({
  perfTrace: v.object({
    enabled: runtimeBoolean,
    token: v.string(),
  }),
  public: v.object({
    posthog: v.object({
      enabled: runtimeBoolean,
      replaySampleRate: runtimeNumber,
    }),
  }),
})
```

Reference the module from `nuxt.config.ts`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  safeRuntimeConfig: {
    $schema: './runtime-config.schema',
    validateAtRuntime: true,
  },
})
```

The module executes the original Standard Schema once after Nitro applies runtime overrides. Asynchronous schemas finish before request handlers run, server-side `useSafeRuntimeConfig()` returns `result.value`, and Nuxt serializes transformed `public` values into the client payload.

Nitro startup plugins run synchronously. Use a synchronous schema when reading `useSafeRuntimeConfig()` during plugin initialization; asynchronous schemas are ready from request handlers and Nuxt rendering onward.

Consumers can use the canonical output directly:

```ts [server/plugins/perf-trace.ts]
const config = useSafeRuntimeConfig()

if (config.perfTrace.enabled)
  startTracing()
```

Keep parsing and normalization inside the schema module. Consumer code should not accept both raw and transformed types or repeat checks such as `value === true || value === 'true'`.

Shelve runtime fetching remains a separate integration. When `shelve.fetchAtRuntime` is enabled, that runtime fetch plugin runs before validation so fetched secrets can be validated.

What it catches:

- Missing env vars in production (`DATABASE_URL` not set)
- Type mismatches (`NUXT_PORT=abc` when expecting number)
- Invalid values at runtime

## Error Handling

Use `onError` to control how both build-time and runtime validation errors are handled:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  safeRuntimeConfig: {
    $schema: runtimeConfigSchema,
    onError: 'throw', // 'throw' | 'warn' | 'ignore'
  },
})
```

- **`'throw'`** (default) - stops the build; runtime validation rejects requests, and synchronous schemas also stop server startup
- **`'warn'`** - logs warning, continues
- **`'ignore'`** - silent

## Error Messages

When validation fails:

```
[safe-runtime-config] Validation failed!
  1. databaseUrl: Invalid type: Expected string but received undefined
  2. public.apiBase: Invalid type: Expected string but received undefined
  3. port: Invalid type: Expected number but received string
```

## Both Modes

Enable both modes when you want build-time checks and runtime transformation:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  safeRuntimeConfig: {
    $schema: './runtime-config.schema',
    validateAtBuild: true, // catch dev errors
    validateAtRuntime: true, // catch prod errors
    onError: 'throw',
  },
})
```
